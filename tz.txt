Автоматизация тестирования REST API

Цель работы: Изучение принципов функционирования программных интерфейсов (API) и получение практических навыков проектирования и реализации автоматизированных тестов для проверки корректности работы REST-сервисов.

Основные задачи:

2.	Освоить функциональные возможности инструмента Postman для выполнения ручных запросов и настройки сред тестирования с использованием переменных окружения (Environments).
3.	Приобрести практические навыки написания автоматизированных тест-скриптов для автоматической проверки корректности данных, типов полей и бизнес-логики API.
4.	Закрепить навыки разработки автоматизированных коллекций для верификации индивидуальных API-ресурсов, обеспечив полное покрытие как позитивных, так и негативных сценариев обработки ошибок.

2.1 Теоретические сведения
Postman — это полноценная API-платформа для разработчиков, которая охватывает весь жизненный цикл создания API: от проектирования и тестирования до документирования и мониторинга. 
Основные функции Postman:
Разработка и отладка (API Client): можно отправлять любые HTTP-запросы (GET, POST, PUT, DELETE) на свой сервер, настраивать заголовки, параметры и тело запроса в удобном интерфейсе, не написав ни строчки кода на фронтенде.
•	Автоматизация тестирования (Testing): В Postman можно писать скрипты на JavaScript, которые будут автоматически проверять ответы сервера. Например: «равен ли статус-код 200?» или «содержит ли JSON поле 'id'?». Это позволяет проводить регрессионное тестирование.
•	Документирование API (Documentation): Postman умеет автоматически генерировать красивую и понятную документацию на основе запросов (Collections). Другие разработчики смогут просто открыть ссылку и увидеть, какие эндпоинты есть у сервера и как с ними работать.
•	Имитация сервера (Mock Servers): Вы можете создать «заглушку» — виртуальный сервер, который будет отдавать заранее прописанные ответы. Это полезно, когда бэкенд ещё не готов, но фронтенд-разработчикам уже нужно с чем-то работать.
•	Командная работа (Collaboration): Postman позволяет создавать «рабочие пространства» (Workspaces), где команда может совместно редактировать запросы, делиться окружениями (например, переключаться между локальным сервером и боевым одним кликом) и оставлять комментарии.
Postman предлагает две версии — Desktop App (приложение для компьютера) и Web Dashboard (версия в браузере). 
1.	Версия для рабочего стола (Desktop App):
Это полноценная программа, которую скачивается и устанавливается на ПК. Она обязательна, если программа разрабатывается локально.
2. Веб-версия (Web Dashboard) — Для совместной работы и быстрого доступа. Она открывается прямо в браузере (Chrome, Firefox и т.д.) по адресу go.postman.co.
Для лабораторной работы будет использоваться Desktop App.
Настройка сервера.

В лабораторной работе будет проводится тестирование взаимодействия (Клиент ↔ Сервер), проверка внутренней логики и имитация сетевого обмена. 
Для тестирования взаимодействие серверной и клиентской части, необходимо поднять и настроить сервер. Для описанного раннее примера функции processOrder был выбран, установлен и настроен сервер Express.
Установка Express:
npm install express

После установки создается файл server.js в той же папке, где лежит orderProcessor.js. Также нужно создать эндпоинт (например, POST /api/order), который будет принимать JSON.
Так как изначально функция требует объект services (курс валют, запись в журнал логов), имитируем их работу внутри сервера.

const express = require('express');
const processOrder = require('./orderProcessor'); 

const app = express();
app.use(express.json()); 

const PORT = 3000;

// Имитация сервисов 
const services = {
  currencyService: {
    getRate: async (code) => {
      console.log(`[Service] Получение курса для: ${code}`);
      return 1.0; // Курс 1 к 1
    }
  },
  loggerService: {
    logOrder: (orderResult) => {
      console.log('[Service] Запись в журнал:', orderResult);
    }
  }
};

// Создаем эндпоинт для обработки заказа
app.post('/api/order', async (req, res) => {
  try {
    const { items, userBalance, minOrderSum } = req.body;

    // Вызываем вашу функцию
    const result = await processOrder(items, userBalance, services, minOrderSum);

    // Если всё успешно, возвращаем 200 OK и результат
    res.status(200).json(result);
    
  } catch (error) {
    // Если функция выбросила ошибку (Error), возвращаем 400 Bad Request
    res.status(400).json({
      status: 'error',
      message: error.message
    });
  }
});

app.get('/api/status', (req, res) => {
    res.status(200).json({ status: 'online', timestamp: new Date() });
  });

app.listen(PORT, () => {
  console.log(`Сервер запущен на http://localhost:${PORT}`);
  console.log('Готов к тестированию API через Postman');
});

После настройки запускаем сервер:
node server.js

Если появилось сообщение Сервер запущен на http://localhost:3000, то сервер запущен корректно.

// 1. Проверка статус-кода

// 3. Проверка типов данных и схемы
pm.test("Соответствие JSON-схеме", () => {
    pm.expect(jsonData).to.have.all.keys('status', 'total', 'remainingBalance');

    pm.expect(jsonData.status).to.eql('success');

// 4. Валидация математических расчетов
// (10.50*2 + 50*1 + 30*1) * 1.0 = 101
pm.test("Математический расчет верен (Total == 101)", () => {
    pm.expect(jsonData.total).to.eql(101);
    pm.expect(jsonData.remainingBalance).to.eql(399); // 500 - 101
});

2.2 Этапы выполнения лабораторной работы 
1.	Подготовка (Backend):
•	Запустить локальный сервер (например, на Express.js), который содержит логику вашего варианта.
•	Убедиться, что логика обработки ошибок (пустая корзина, отрицательные числа) возвращает соответствующие статус-коды (400 вместо 200).
2.	Создание коллекции в Postman:
•	Создать новую коллекцию (Collection) и добавить в неё запросы для вашего варианта.
•	Настроить переменные окружения (base_url).
3.	Реализация тестов для позитивных сценариев:
•	Написать тесты на проверку успешного расчета.
•	Проверить типы данных в JSON-ответе (например, что total — это число).
4.	Реализация тестов для негативных сценариев:
•	Отправить некорректные данные.
•	Убедиться, что API возвращает ошибку 400 и понятное сообщение (например, "Недостаточно средств").
5.	Сценарий интеграции (Chaining):
•	Реализовать минимум два связанных запроса. Например:
GET /status (проверка готовности сервиса).
POST /calculate (выполнение основного задания).
6.	Запуск через Collection Runner:
•	Запустить всю коллекцию целиком и убедиться, что все тесты (проверки) проходят успешно ("зеленый свет").
Внимание! Итоговые тесты должны содержать следующие проверки:
•	Проверка статус-кода (200, 201, 400).
•	Проверка времени ответа (менее 200 мс).
•	Проверка соответствия JSON-схемы.
•	Валидация математических расчетов.
